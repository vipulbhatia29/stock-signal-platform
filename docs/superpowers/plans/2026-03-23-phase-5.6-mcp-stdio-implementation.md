## S1: MCP Tool Server (stdio) — ~2.5h

**Spec ref:** §4 (MCP Tool Server)
**Branch:** Story branch (created after plan approval)

### Tasks

**T1.1: Create tool server entry point**
- **Create:** `backend/mcp_server/tool_server.py`
- Standalone script that:
  - Imports all 20 tool classes + 4 MCP adapters
  - Creates its own async DB engine + session factory from `DATABASE_URL`
  - Creates its own Redis client from `REDIS_URL`
  - Builds a `ToolRegistry` and registers everything
  - Creates a `FastMCP` server, registers all tools from registry
  - Each tool handler: calls `tool.execute(params)`, serializes `ToolResult` to JSON
  - Runs via `mcp.run(transport="stdio")`
- **Key detail:** Tool server must work as `uv run python -m backend.mcp_server.tool_server`
- **Test:** Manual — run the script, verify it starts and responds to list_tools

**T1.2: Extract registry builder**
- **Create:** `backend/tools/build_registry.py`
- Extract tool + adapter registration from `main.py` lifespan into a reusable `build_registry()` function
- Both `main.py` and `tool_server.py` call this function
- Avoids duplicating the 20-tool import list
- **Modify:** `backend/main.py` — replace inline registration with `build_registry()` call
- **Test:** Existing unit tests pass (no behavior change)

**T1.3: ToolResult JSON serialization**
- **Modify:** `backend/tools/base.py` — add `to_json()` / `from_json()` class methods to `ToolResult`
- `to_json()`: returns `json.dumps({"status": ..., "data": ..., "error": ...})`
- `from_json(text)`: returns `ToolResult` from JSON string
- Handle non-serializable data (datetime, Decimal) via custom encoder
- **Test:** `tests/unit/mcp/test_tool_result_serialization.py` — round-trip tests with various data types

**T1.4: Unit tests for tool server**
- **Create:** `tests/unit/mcp/test_tool_server.py`
- Test that `build_registry()` returns all 20 tools
- Test that tool server registers all tools on FastMCP
- Test ToolResult serialization edge cases (empty data, error, large payloads)

### Definition of Done
- `uv run python -m backend.mcp_server.tool_server` starts without error
- All existing tests pass (registry builder refactor is backward-compatible)
- 6+ new unit tests

## S2: MCP Tool Client — ~2h

**Spec ref:** §5 (MCP Client Integration)

### Tasks

**T2.1: Create MCPToolClient**
- **Create:** `backend/mcp_server/tool_client.py`
- Class `MCPToolClient`:
  - `async connect(server_script: str)` — spawns subprocess via `mcp` SDK's `StdioServerParameters` + `stdio_client()`, establishes `ClientSession`
  - `async call_tool(name: str, params: dict) -> ToolResult` — calls `session.call_tool(name, params)`, deserializes response via `ToolResult.from_json()`
  - `async list_tools() -> list[str]` — calls `session.list_tools()`, returns tool names
  - `async close()` — terminates subprocess (SIGTERM → 5s → SIGKILL)
  - `@property is_connected -> bool` — check if subprocess is alive
- **Key detail:** Use `mcp.client.stdio` module for stdio transport setup
- **Test:** Unit tests with mocked subprocess

**T2.2: User context injection**
- **Modify:** `backend/agents/executor.py` — before calling tool_executor, inject `user_id` into params if `current_user_id` ContextVar is set and tool name is in portfolio tools list
- Portfolio tool names: `portfolio_exposure`, `get_portfolio_forecast`, `recommendations`
- **Modify:** `backend/tools/portfolio_exposure.py` — accept optional `user_id` param, use it instead of ContextVar when provided
- **Test:** Unit test that user_id injection works correctly

**T2.3: Unit tests for client**
- **Create:** `tests/unit/mcp/test_tool_client.py`
- `test_call_tool_deserializes_ok_result` — mock session returns OK JSON → ToolResult
- `test_call_tool_deserializes_error_result` — mock session returns error JSON → ToolResult(status="error")
- `test_call_tool_timeout` — mock session hangs → TimeoutError
- `test_list_tools_returns_names` — mock session returns tool list
- `test_close_terminates_process` — verify cleanup

### Definition of Done
- MCPToolClient can connect, call tools, list tools, and close
- User context injection works for portfolio tools
- 8+ new unit tests

## S3: Lifespan Wiring + Feature Flag — ~2h

**Spec ref:** §6 (Subprocess Lifecycle), §7 (Feature Flag)

### Tasks

**T3.1: Add MCP_TOOLS config**
- **Modify:** `backend/config.py` — add `MCP_TOOLS: bool = True`
- **Modify:** `backend/.env.example` — add `MCP_TOOLS=true` with comment

**T3.2: Create subprocess lifecycle manager**
- **Create:** `backend/mcp_server/lifecycle.py`
- Class `MCPSubprocessManager`:
  - `async start()` — spawns tool server, verifies with `list_tools()`
  - `async restart()` — kills + respawns, increments restart counter
  - `async ensure_healthy()` — checks subprocess alive, triggers restart if dead
  - `async stop()` — graceful shutdown
  - Properties: `healthy: bool`, `restart_count: int`, `mode: str` ("stdio" | "fallback_direct" | "disabled")
  - Constants: `MAX_RESTARTS = 3`
- On max restarts exceeded: sets `mode = "fallback_direct"`, logs ERROR
- **Test:** Unit tests with mocked MCPToolClient

**T3.3: Wire into FastAPI lifespan**
- **Modify:** `backend/main.py`
  - If `settings.MCP_TOOLS`:
    - Create `MCPSubprocessManager`, call `start()`
    - Build `_tool_executor` that calls `mcp_client.call_tool()`
    - On subprocess failure (caught in executor): call `ensure_healthy()`, retry once, then fall back to `registry.execute()`
    - Store manager on `app.state.mcp_manager`
  - Else:
    - `_tool_executor` calls `registry.execute()` (current behavior)
  - Shutdown: call `manager.stop()`
- **Modify:** `backend/agents/graph_v2.py` — accept `tool_executor` callable (already does this), no changes needed

**T3.4: Unit tests for lifecycle**
- **Create:** `tests/unit/mcp/test_lifecycle.py`
- `test_start_spawns_subprocess` — verify connect called
- `test_restart_increments_counter` — verify count tracking
- `test_max_restarts_triggers_fallback` — after 3 failures, mode = fallback_direct
- `test_ensure_healthy_restarts_dead_process` — dead process → restart
- `test_stop_calls_close` — verify cleanup

### Definition of Done
- `MCP_TOOLS=true` spawns subprocess in lifespan, agent uses MCP client
- `MCP_TOOLS=false` uses direct registry calls (current behavior)
- Fallback to direct after 3 restart failures
- 8+ new unit tests

## S4: Health Endpoint + Observability — ~1.5h

**Spec ref:** §8 (Health Endpoint)

### Tasks

**T4.1: Create or extend health endpoint**
- **Create or modify:** `backend/routers/health.py`
- `GET /api/v1/health` — public endpoint (no auth required)
- Response schema (Spec §8.1):
  ```json
  {
    "status": "ok" | "degraded",
    "version": "0.1.0",
    "mcp_tools": {
      "enabled": bool,
      "mode": "stdio" | "fallback_direct" | "direct",
      "healthy": bool,
      "tool_count": int,
      "restarts": int,
      "uptime_seconds": float | null,
      "last_error": str | null,
      "fallback_since": str | null
    }
  }
  ```
- Reads from `app.state.mcp_manager` (if MCP_TOOLS enabled) or returns static "direct" mode

## Rollback Plan

If the MCP stdio path causes issues after merging to develop:
1. Set `MCP_TOOLS=false` in `.env` → immediate rollback to direct calls
2. Restart the app → subprocess not spawned, agent uses registry directly
3. All existing functionality preserved (Celery, /mcp endpoint, tools unchanged)
