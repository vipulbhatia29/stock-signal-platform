## 1. Problem Statement

The agent currently calls tools via direct in-process Python calls:

```
Agent Executor → ToolRegistry.execute(name, params) → Tool.execute(params) → ToolResult
```

This works but means:
- Tools are tightly coupled to the agent process
- External clients (mobile, Telegram, Slack) must reimplement tool discovery and calling
- No path to independent scaling of tool execution
- Adding new tools requires redeploying the entire app

## 2. Goal

Refactor the agent to consume tools via MCP protocol (stdio transport), establishing the MCP abstraction so:
1. Any future client can consume tools via a standard protocol
2. Phase 6 swaps stdio → Streamable HTTP with zero tool/schema changes
3. Tools can be developed and tested independently

## 3. Architecture Overview

### 3.1 Current (Monolith)

```
┌─────────────────────────────────────────────────────────────┐
│ FastAPI Process                                              │
│                                                              │
│  Chat Router → Agent Graph → Executor → ToolRegistry         │
│                                             ↓                │
│                                     Tool.execute()           │
│                                             ↓                │
│                                     DB / Redis / APIs        │
│                                                              │
│  /mcp (Streamable HTTP) → ToolRegistry → Tool.execute()     │
│  Celery Worker → ToolRegistry → Tool.execute()               │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Target (stdio MCP)

```
┌─────────────────────────────────────────────────────────────┐
│ FastAPI Process                                              │
│                                                              │
│  Chat Router → Agent Graph → Executor → MCPToolClient        │
│                                             ↓ stdio          │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ MCP Tool Server (subprocess)                         │    │
│  │   FastMCP → ToolRegistry → Tool.execute()            │    │
│  │   Own DB pool, own Redis client                      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  /mcp (Streamable HTTP) → ToolRegistry → Tool.execute()     │
│  Celery Worker → ToolRegistry → Tool.execute()               │
│  Health: GET /api/v1/health → subprocess status              │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 What Changes vs. What Stays

| Component | Changes? | Details |
|-----------|----------|---------|
| Tool classes (BaseTool, 20 tools) | **No** | Same execute() interface |
| ToolRegistry | **No** | Same registration + discovery |
| MCP adapters (4 external) | **No** | Same ProxiedTool pattern |
| Agent Planner (LLM) | **No** | Still generates tool plans |
| Agent Synthesizer (LLM) | **No** | Still consumes ToolResult |
| **Agent Executor** | **Yes** | Calls MCP client instead of registry.execute() |
| **Lifespan** | **Yes** | Spawns subprocess, manages lifecycle |
| **Health endpoint** | **Yes** | Reports MCP subprocess status |
| `/mcp` Streamable HTTP | **No** | Still serves external clients directly |
| Celery tasks | **No** | Still call registry directly (no MCP overhead) |
| `backend/config.py` | **Yes** | New `MCP_TOOLS` flag |

## 4. MCP Tool Server

### Terminology Note

| Term | What | Used where |
|------|------|-----------|
| **FastAPI** | Web framework for REST APIs | Parent process only — serves `/api/v1/*`, manages lifespan |
| **FastMCP** | MCP server framework (part of `mcp` Python SDK) | Subprocess only — serves tools via stdio protocol |

The subprocess has **no FastAPI**. It is a pure FastMCP stdio server. FastAPI only appears in the parent process for lifecycle management (spawning/killing the subprocess).

### 4.1 Entry Point

**File:** `backend/mcp_server/tool_server.py`

A standalone script that:
1. Imports all tool classes + adapters
2. Creates a ToolRegistry and registers everything
3. Creates a FastMCP server and registers all tools
4. Runs via `mcp.run(transport="stdio")`

```python
# backend/mcp_server/tool_server.py
"""MCP Tool Server — stdio transport.

Standalone process that exposes all 20+ tools via MCP protocol.
Spawned by FastAPI lifespan, communicates via stdin/stdout.
"""
import asyncio
import json
import logging

from mcp.server.fastmcp import FastMCP

# ... tool imports, registry setup ...

mcp = FastMCP("StockSignal Tools")

def register_all_tools(mcp: FastMCP, registry: ToolRegistry) -> None:
    """Register all tools from registry as MCP tools."""
    for tool_info in registry.discover():
        tool = registry.get(tool_info.name)
        # Each MCP tool handler calls tool.execute() and returns ToolResult as JSON
        @mcp.tool(name=tool_info.name, description=tool_info.description)
        async def handler(params: dict = {}, _tool=tool) -> str:
            result = await _tool.execute(params)
            return json.dumps({"status": result.status, "data": result.data, "error": result.error})

if __name__ == "__main__":
    registry = build_registry()  # Creates ToolRegistry with all tools + adapters
    register_all_tools(mcp, registry)
    mcp.run(transport="stdio")
```

### 4.2 Database & Redis

The subprocess creates its own connections on startup:

```python
# Inside tool_server.py
from backend.config import settings
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker

engine = create_async_engine(settings.DATABASE_URL, pool_size=5)
async_session_factory = async_sessionmaker(engine, expire_on_commit=False)

# Redis for caching
import redis.asyncio as aioredis
redis_client = aioredis.from_url(settings.REDIS_URL, decode_responses=True)
```

Tools that need DB sessions will receive them from the subprocess's session factory, not the main app's.

### 4.3 Tool Result Serialization

MCP tool responses are text content. We serialize the full `ToolResult` as JSON:

```json
{"status": "ok", "data": {"ticker": "AAPL", "price": 178.50, ...}, "error": null}
```

The MCP client deserializes this back into a `ToolResult` object. This preserves the full status/data/error contract across the stdio boundary.

## 5. MCP Client Integration

### 5.1 MCPToolClient

**File:** `backend/mcp_server/tool_client.py`

A thin wrapper around the official `mcp` Python SDK client:

```python
class MCPToolClient:
    """Client that calls tools via MCP stdio transport."""

    async def connect(self, server_script: str) -> None:
        """Spawn the MCP tool server subprocess and establish stdio connection."""

    async def call_tool(self, name: str, params: dict) -> ToolResult:
        """Call a tool via MCP and deserialize the ToolResult."""

    async def list_tools(self) -> list[str]:
        """List available tools (used for health checks)."""

    async def close(self) -> None:
        """Terminate the subprocess gracefully."""
```

### 5.2 Agent Executor Integration

The executor's `tool_executor` callable changes based on `MCP_TOOLS`:

```python
# In main.py lifespan
if settings.MCP_TOOLS:
    mcp_client = MCPToolClient()
    await mcp_client.connect("backend/mcp_server/tool_server.py")
    app.state.mcp_client = mcp_client

    async def _tool_executor(name: str, params: dict) -> ToolResult:
        return await mcp_client.call_tool(name, params)
else:
    async def _tool_executor(name: str, params: dict) -> ToolResult:
        return await registry.execute(name, params)
```

The planner, synthesizer, result validator, and stream events are completely unchanged.

### 5.3 User Context (Portfolio Tools)

`PortfolioExposureTool` and other user-specific tools currently read `current_user_id` from a ContextVar. Since ContextVars don't cross process boundaries, we pass `user_id` as an explicit param:

```python
# In the executor, before calling MCP:
if "user_id" not in params and current_user_id_var.get(None):
    params["user_id"] = str(current_user_id_var.get())
```

Tools that need user context will accept an optional `user_id` param. No changes needed for tools that don't use user context.

## 6. Subprocess Lifecycle Management

### 6.1 Startup

In FastAPI lifespan (after registry setup):

```python
if settings.MCP_TOOLS:
    mcp_client = MCPToolClient()
    await mcp_client.connect("backend/mcp_server/tool_server.py")
    # Verify health
    tools = await mcp_client.list_tools()
    logger.info("MCP Tool Server started with %d tools", len(tools))
    app.state.mcp_client = mcp_client
    app.state.mcp_healthy = True
```

### 6.2 Health Checks

Periodic `list_tools()` call (via background task or on first request after interval):

```python
async def check_mcp_health() -> bool:
    try:
        tools = await asyncio.wait_for(mcp_client.list_tools(), timeout=5.0)
        return len(tools) > 0
    except Exception:
        return False
```

### 6.3 Restart Policy

| Event | Action |
|-------|--------|
| Subprocess exits unexpectedly | Log WARNING, attempt restart |
| Restart succeeds | Reset restart counter, continue |
| 3 consecutive restart failures | Log ERROR, set `app.state.mcp_healthy = False`, fall back to direct calls |
| Direct fallback active | Continue serving, health endpoint reports degraded |

### 6.4 Shutdown

```python
# In lifespan cleanup
if hasattr(app.state, "mcp_client"):
    await app.state.mcp_client.close()  # SIGTERM → wait 5s → SIGKILL
```

## 9. Error Handling

### 9.1 Error Flow Across stdio Boundary

```
Tool raises exception
  → Tool.execute() catches, returns ToolResult(status="error", error="message")
  → MCP server serializes to JSON text content
  → stdio pipe
  → MCP client deserializes back to ToolResult
  → Executor's validate_tool_result() annotates as "unavailable"
  → Synthesizer handles gracefully
```

### 9.2 MCP Transport Errors

| Error | Handling |
|-------|----------|
| Subprocess not responding (timeout) | Client raises TimeoutError → executor treats as tool timeout |
| Subprocess crashed (broken pipe) | Client raises ConnectionError → trigger restart → retry tool call once |
| Invalid JSON from subprocess | Client raises ParseError → treat as tool error |
| Unknown tool name | MCP server returns error → client maps to ToolResult(status="error") |

### 9.3 No Stack Traces to Frontend

All errors are caught at the MCP boundary. The agent stream only emits sanitized `tool_error` events, same as today.

## 10. Caching Strategy

No changes to caching logic. The subprocess has its own Redis connection and uses the same cache keys + TTLs as the current in-process tools. Since both processes connect to the same Redis instance, cache entries are shared automatically.

## 12. Migration Path to Phase 6 (Streamable HTTP)

Phase 5.6 establishes the MCP abstraction. Phase 6 changes only the transport:

| Aspect | Phase 5.6 (stdio) | Phase 6 (Streamable HTTP) |
|--------|-------------------|---------------------------|
| Transport | stdio pipes | HTTP (port 8282) |
| Process model | Subprocess | Separate container |
| MCPToolClient config | `server_script` path | `server_url` URL |
| Tool schemas | Identical | Identical |
| ToolResult format | Identical | Identical |
| DB connection | Own pool | Own pool |
| Redis | Own client | Own client |

The only code change in Phase 6 is in `MCPToolClient.connect()`:
```python
# Phase 5.6
await client.connect(server_script="backend/mcp_server/tool_server.py")

# Phase 6
await client.connect(server_url="http://mcp-tools:8282")
```

## 15. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| stdio serialization overhead | Low | Low | JSON round-trip adds ~1ms. Measure in integration tests. |
| Subprocess memory leak | Low | Medium | Health checks detect unresponsive process → auto-restart |
| DB pool exhaustion (two pools) | Low | High | Subprocess pool_size=5 (small), main app unchanged |
| Stale tool list after registry change | Low | Low | Static registration — restart subprocess to pick up changes |
| ContextVar loss across process boundary | Certain | Medium | Mitigated: pass user_id as explicit param |
| CI flakiness from subprocess lifecycle | Medium | Low | Retry logic in test fixtures, generous timeouts |
