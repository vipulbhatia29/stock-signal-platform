# BU-5: Observability Backend API Gaps — Spec

**JIRA:** KAN-231 (Story under Epic KAN-226 Phase B.5)
**Date:** 2026-03-30
**Status:** Draft
**Depends on:** BU-1 (schema types — DONE)
**Blocks:** BU-6 (Observability Frontend)

## 1. Problem Statement

The observability backend has 6 endpoints (KPIs, query list, query detail, Langfuse URL, assessment latest/history) but the query list and detail endpoints have gaps that prevent the BU-6 frontend from being fully functional:

1. **No server-side sorting** — query list hardcodes `ORDER BY timestamp DESC`
2. **No status filter** — status is hardcoded to `"completed"` in the response
3. **No cost threshold filter** — can't find expensive queries
4. **No group-by aggregation** — can't build charts or summary views
5. **Input/output summaries missing** — step detail always returns `None` for summaries
6. **Eval score never populated** — `QueryRow.score` is always `None`

Additionally, two cross-cutting concerns surfaced during brainstorming:
- **PII in tool summaries** — admins should not see user identifiers in tool params/results
- **Langfuse deep-link** — `langfuse_trace_url` always returns `None`; needs trace ID propagation

## 2. Approach

**Incremental enhancement** of the existing service layer (`observability_queries.py`). No new service files, no abstraction layers. One migration (DB is empty — no backfill). One new endpoint for group-by. Instrumentation changes in the single write point (`observability_writer.py`).

| `output_summary` | `TEXT` | YES | NULL | Truncated, PII-sanitized tool result (≤300 chars) |

### 3.3 `eval_results` (AssessmentResult)

| Column | Type | Nullable | Default | Purpose |
|--------|------|----------|---------|---------|
| `query_id` | `UUID` | YES | NULL | Logical reference to `llm_call_log.query_id` for eval score join |

**Index:** `CREATE INDEX ix_eval_results_query_id ON eval_results (query_id)` — supports the LEFT JOIN.

**Note:** `query_id` is not a foreign key. `llm_call_log` is a TimescaleDB hypertable and `query_id` is a logical grouping key (not unique). The index enables efficient JOIN.

**Migration must be hand-written** (not autogenerated) — Alembic autogenerate falsely detects all TimescaleDB hypertables as new tables and rewrites the entire schema. Write incremental `op.add_column()` calls only.

## 4. Enhanced Query List Endpoint

**`GET /api/v1/observability/queries`**

### 4.1 New Query Parameters

| Param | Type | Default | Validation |
|-------|------|---------|------------|
| `sort_by` | `SortByEnum` | `timestamp` | `timestamp \| total_cost_usd \| duration_ms \| llm_calls \| score` |
| `sort_order` | `SortOrderEnum` | `desc` | `asc \| desc` |
| `status` | `str \| None` | `None` | `completed \| error \| declined \| timeout` |
| `cost_min` | `float \| None` | `None` | `ge=0` |
| `cost_max` | `float \| None` | `None` | `ge=0` |

Existing params unchanged: `page`, `size`, `agent_type`, `date_from`, `date_to`.

### 4.2 Sort Implementation

Sort columns map to SQL expressions on the aggregated subquery:

| `sort_by` value | SQL expression |
|-----------------|----------------|
| `timestamp` | `MIN(llm_call_log.created_at)` |
| `total_cost_usd` | `SUM(llm_call_log.cost_usd)` |
| `duration_ms` | Requires a correlated subquery: `(SELECT SUM(latency_ms) FROM tool_execution_log t WHERE t.query_id = llm_call_log.query_id)` added as a column in the base grouped query via `lateral` or scalar subquery. **Non-trivial** — this subquery must be part of the base SELECT so it is available for ORDER BY before pagination. |
| `llm_calls` | `COUNT(*)` |
| `score` | eval score (LEFT JOIN), `NULLS LAST` |

Sort is whitelist-validated via enum — no raw column interpolation in SQL.

### 4.3 Status Filter

**Derived status per query group:** Each query group derives its worst status via `MAX(CASE WHEN status = 'error' THEN 3 WHEN status = 'declined' THEN 2 WHEN status = 'timeout' THEN 1 ELSE 0 END)` — if any call errored, the query status is `error`.

**Filter semantics:** The `status` query parameter filters on the **derived** group status, not on individual LLM call rows. This is implemented as a `HAVING` clause on the derived status expression (post-aggregation), not a `WHERE` clause (pre-aggregation). This ensures `status=error` returns queries that had at least one error, not just the error rows within a query.

**Example:** A query with 3 LLM calls (2 completed, 1 error) has derived status `error`. Filtering `status=error` includes this query. Filtering `status=completed` excludes it.

### 4.4 Cost Filter

- Applied as `HAVING` clauses on the grouped query:
  - `HAVING SUM(cost_usd) >= :cost_min` (when `cost_min` provided)
  - `HAVING SUM(cost_usd) <= :cost_max` (when `cost_max` provided)
- **Pagination interaction:** HAVING clauses (both cost and status) must be applied to the `base` query **before** the count subquery is constructed. Otherwise `total` will over-count by including groups that are filtered out by HAVING.
- **Validation:** If `cost_min > cost_max`, return 422 via a Pydantic `model_validator` or explicit check in the router — do not silently return empty results.

### 4.5 Eval Score Join

- LEFT JOIN `eval_results` on `eval_results.query_id = llm_call_log.query_id`
- Composite score: `CASE WHEN reasoning_coherence_score IS NOT NULL THEN (grounding_score + reasoning_coherence_score) / 2 ELSE grounding_score END`
- **Design note:** Only `grounding_score` and `reasoning_coherence_score` are used because they are continuous floats (0.0–1.0). The other 3 assessment dimensions (`tool_selection_pass`, `termination_pass`, `external_resilience_pass`) are booleans — mixing pass/fail into an average produces misleading scores. The boolean dimensions are visible in the assessment detail view (BU-6).
- `QueryRow.score` populated when join matches, `None` for unassessed queries
- Only assessment runner queries (which propagate `query_id` into `eval_results`) will have scores

## 5. Enhanced Query Detail Endpoint

**`GET /api/v1/observability/queries/{query_id}`**

### 5.1 Input/Output Summaries

**Tool steps:**
- `StepDetail.input_summary` — read from `ToolExecutionLog.input_summary` (stored at write time)
- `StepDetail.output_summary` — read from `ToolExecutionLog.output_summary` (stored at write time)

**LLM steps (derived at read time):**
- `StepDetail.input_summary` — `"→ {provider}/{model}"`
- `StepDetail.output_summary` — `"{completion_tokens} tokens, {latency_ms}ms, ${cost_usd}"`

### 5.2 Langfuse Deep-Link

- Read `langfuse_trace_id` from the first LLM call in the query group
- Construct URL: `{settings.LANGFUSE_BASEURL}/trace/{langfuse_trace_id}`
- Guard: if `LANGFUSE_BASEURL` is `None`/empty or `langfuse_trace_id` is `None`, return `langfuse_trace_url: None` — do not produce malformed URLs
- `QueryDetailResponse.langfuse_trace_url` populated only when both trace ID and host are present
- `LANGFUSE_BASEURL` setting already exists in `backend/config.py` (line 76, used by Langfuse SDK and existing `langfuse_url` endpoint)

## 6. Group-By Endpoint (New)

**`GET /api/v1/observability/queries/grouped`**

**Route registration order:** This endpoint MUST be registered in `observability.py` BEFORE the existing `/queries/{query_id}` route. FastAPI matches routes in registration order — if `{query_id}` is first, `"grouped"` will be parsed as a UUID and return 422 instead of reaching this handler.

### 6.1 Parameters

| Param | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `group_by` | `GroupByEnum` | YES | — | Grouping dimension (see §6.2) |
| `date_from` | `datetime \| None` | NO | None | Filter window start |
| `date_to` | `datetime \| None` | NO | None | Filter window end |
| `bucket` | `DateBucketEnum` | NO | `day` | Only for `group_by=date`: `day \| week \| month` |

### 6.2 Supported Groups (9)

| Group | Source Table | Key Value | Notes |
|-------|-------------|-----------|-------|
| `agent_type` | `llm_call_log` | e.g., `"react_v2"`, `"stock"` | |
| `date` | `llm_call_log` | e.g., `"2026-03-28"` | Uses `date_trunc(:bucket, created_at)` — leverages TimescaleDB |
| `model` | `llm_call_log` | e.g., `"llama-3.3-70b"` | |
| `status` | `llm_call_log` | `"completed"`, `"error"`, etc. | |
| `provider` | `llm_call_log` | `"groq"`, `"anthropic"`, `"openai"` | |
| `tier` | `llm_call_log` | `"fast"`, `"balanced"`, `"powerful"` | |
| `tool_name` | `tool_execution_log` | e.g., `"analyze_stock"` | Different base table |
| `user` | `llm_call_log` JOIN `chat_sessions` JOIN `users` | User email | **Admin-only** — 403 for non-admins |
| `intent_category` | `eval_results` | e.g., `"stock_analysis"` | Only assessed queries; may return empty. **No user scoping** — assessment runs are admin-generated data, so all users see the same results for this dimension |

### 6.3 Response Schema

```python
class GroupRow(BaseModel):
    key: str
    query_count: int
    total_cost_usd: float
    avg_cost_usd: float
    avg_latency_ms: float
    error_rate: float       # 0.0–1.0

class GroupedResponse(BaseModel):
    group_by: str
    bucket: str | None = None
    groups: list[GroupRow]
    total_queries: int      # sum across all groups
```

### 6.4 Security

- `group_by=user` checks admin role before executing — raise `HTTPException(403)` for non-admins
- **Admin guard extraction:** `_require_admin()` currently lives in `backend/routers/admin.py`. Extract it to `backend/dependencies.py` as a shared dependency so both admin and observability routers can import it without cross-router coupling.
- All other groups use `_user_scope(user)` — regular users see only their own data aggregated
- Exception: `intent_category` has no user scoping (assessment data is admin-generated, see §6.2)
- `group_by` is an enum, never interpolated into SQL
- **Date group key serialization:** `date_trunc()` returns a `timestamp` — serialize to ISO 8601 string (e.g., `"2026-03-28T00:00:00"`) when building `GroupRow.key` to satisfy Pydantic `str` type

## 7. PII Sanitization

**New utility: `backend/utils/sanitize.py`**

### 7.1 `sanitize_summary(raw: Any, max_length: int = 300) -> str`

Applied at write time before storing `input_summary` and `output_summary`.

**Input handling:** The function accepts `Any` (dict, list, str, or other). If `raw` is a dict or list, it is processed structurally (recursive key walk). If it is a string, it is first attempted as `json.loads()` to recover structure; if that fails, it is treated as plain text (only email regex applies).

**Sanitization rules (applied in order):**
1. **Field blocklist (recursive)** — Walk the dict/list structure recursively. Any key matching `user_id`, `email`, `password`, `token`, `api_key`, `secret`, `authorization` → replace its value with `[REDACTED]`. Applies at all nesting levels.
2. **Email regex** — `\b[\w.+-]+@[\w-]+\.[\w.-]+\b` → `[EMAIL]` (applied to the final serialized string)
3. **JSON serialize** — `json.dumps()` the sanitized structure
4. **Truncation** — slice to `max_length` characters

**What is NOT sanitized:**
- Ticker symbols, prices, dates, financial data — this is operational signal admins need
- Tool names, status codes, latency values

### 7.2 Error Handling

- If JSON serialization fails (non-serializable input), fall back to `str(x)[:max_length]`
- If sanitization itself fails, return `"[SANITIZE_ERROR]"` — never propagate exceptions from summary generation

## 8. Instrumentation

**Single write point: `backend/agents/observability_writer.py` → `write_event()`**

### 8.1 Tool Execution Events

**Note:** `ToolExecutionLog.params` (JSONB) continues to be written unchanged for internal debugging. `input_summary` is the sanitized, truncated display column for the observability API. They serve different purposes and both are written.

**Caller changes (2 files):**
1. **`backend/agents/react_loop.py`** (line ~255) — the ReAct loop is the primary tool executor (NOT `executor.py` which is the V1 executor). It already has `result.data` available after tool execution. Pass `result=result.data` to `collector.record_tool_execution()`.
2. **`backend/agents/observability.py`** — update `ObservabilityCollector.record_tool_execution()` signature to accept a `result: Any = None` parameter and include it in the `data` dict passed to `write_event()`.

**Note:** `backend/agents/executor.py` is the V1 executor (pre-ReAct, feature-flagged off via `AGENT_V2`). It also calls `collector.record_tool_execution()` (line ~257). Update it for consistency, but the V2 ReAct loop in `react_loop.py` is the active code path.

### 8.2 LLM Call Events

Add to `LLMCallLog(...)` constructor:
- `status=data.get("status", "completed")` — set to `"error"` when `data["error"]` is present
- `langfuse_trace_id=data.get("langfuse_trace_id")`

**Caller changes:**
- **`backend/agents/llm_client.py` → `LLMClient.chat()`**: Already calls `self._langfuse.get_trace_ref(qid)` (line 248) using `current_query_id` from `backend/request_context.py`. The `qid` IS the trace ID (Langfuse uses it as `trace(id=str(trace_id))`). So `langfuse_trace_id = current_query_id.get(None)` — no new ContextVar needed. Pass `langfuse_trace_id=str(qid)` in the data dict to `collector.record_request()`.
- **`backend/agents/observability.py` → `ObservabilityCollector.record_request()`**: Add `langfuse_trace_id` param, include in data dict passed to `write_event()`.
- **Chat router decline paths (`backend/routers/chat.py`)**: There are 3 decline paths that currently do NOT write to `llm_call_log`:
  1. Input guard failure — `validate_input_length()` (line 109-111)
  2. Injection detection — `detect_injection()` (line 131-142)
  3. Intent classifier `out_of_scope` — (line 237-248)
  Each decline path should call `collector.record_request(..., status="declined")` before returning the decline stream, so declined queries appear in observability data. The collector is available in the chat endpoint scope — instantiate one at the top of the endpoint function.
- **Note:** `guards.py` contains only validation functions (pure logic). The decline flow is orchestrated in the chat router, not in guards.

### 8.3 Assessment Runner

When executing golden dataset queries:
- **ContextVar setup:** The assessment runner's `_run_query_live()` invokes the ReAct loop directly (not via the chat router). It must explicitly set `current_query_id` ContextVar before invoking the loop — currently this ContextVar is only set in the chat router path. Add `current_query_id.set(query_id)` at the start of `_run_query_live()`.
- Propagate `query_id` into `AssessmentResult.query_id` when creating result rows in the `AssessmentResult(...)` constructor call
- This enables the eval score LEFT JOIN in §4.5
